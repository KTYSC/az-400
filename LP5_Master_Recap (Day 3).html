⚙️ LP5 — Infrastructure as Code
🎯
LP5 Throughline: Every manual step is a risk. Every undocumented change is a future incident. IaC is how you eliminate both.
The Problem — Manual Infrastructure

Why IaC Exists

  • Snowflake servers — each built slightly differently by different engineers over time
  • Deployment steps vary per environment — dev ≠ test ≠ prod
  • More verification steps, more elaborate manual processes
  • Deploy on weekends to allow recovery time from errors
  • Slower release cadence to minimise risk
  • When one breaks — nobody knows exactly what's on it
🔥
Snowflake problem: each server becomes unique over time. When it breaks, it's irreproducible.
The Solution — Infrastructure as Code

What IaC Gives You

  • Consistent servers between environments
  • Environments created or scaled easily from one file
  • Fully automated creation and updates
  • Immutable infrastructure — don't patch, redeploy
  • Blue/green deployments with zero downtime
  • Treat servers as commodities, not pets
💡
Your Bicep file IS the documentation. It can't drift from reality the way a Word doc can.

EXAM: ARM, Bicep, DSC = declarative. Azure CLI = imperative. This distinction appears in idempotency questions. Know it cold.
Idempotency — Run It 10 Times, Same Result

❌ Non-Idempotent (Bash Script)

Run 1 → creates storage account ✓ Run 2 → tries to create again → ERROR Run 3 → ERROR again

Script isn't idempotent. Every run after the first fails. Pipelines re-run. Retries fire. This breaks.

✅ Idempotent (ARM / Bicep)

Run 1 → creates storage account ✓ Run 2 → already exists, nothing changes ✓ Run 3 → still compliant ✓

Declarative tools check state first. If already correct — nothing happens. Safe to run repeatedly.

💜
Expert Insight: Idempotency is the operating model. Something drifts? Don't patch it. Discard it and redeploy from the template. The template is always the source of truth.
📌
Achieve Idempotence Two Ways: (1) Auto-configure and reconfigure existing resources — OR — (2) Discard existing resources and spin up a fresh environment.

Managing Dependencies — dependsOn

ARM deploys in parallel by default — fast, but breaks when order matters.

virtualNetwork (no dependencies) ↑ networkInterface dependsOn: virtualNetwork ↑ virtualMachine dependsOn: networkInterface storageAccount

Two ways to declare dependencies:

  • Explicit — use the dependsOn element
  • Implicit — use reference() function inside a resource; ARM infers the dependency automatically
⚠️
Circular References: A depends on B, B depends on A → deployment fails. Fix by breaking the cycle or moving a dependency into a child resource.
Deployment Modes — Exam Critical
Mode Behaviour
Incremental (default) Deploys what's in the template. Leaves everything else in the RG alone. Safe.
Complete Deploys what's in the template. DELETES anything in the RG NOT in the template. Destructive.
🎯
EXAM: Incremental = deploy what's in template, leave the rest alone. Complete = delete anything NOT in template. Incremental is almost always correct.
💡
Expert: Use one resource group per deployment as best practice. Reduces blast radius if Complete mode is accidentally used.
Modularizing Templates — Linked Templates
main.json (orchestrator) ├── linked: network.json ├── linked: storage.json └── linked: compute.json Each template: → single responsibility → reusable across deployments → independently testable

Linked templates referenced by URL — must be accessible at deploy time. Usually stored in a storage account or public GitHub URL.

Benefits

  • Linked templates return outputs back up to main
  • Each component can be tested independently
  • Reuse network template across multiple projects

Bicep = ARM with less noise. Same deployment engine, same resource types, same idempotency. Dramatically less code. Microsoft's recommended approach for new Azure IaC today.
ARM vs Bicep — Same Storage Account

ARM JSON (~200 lines)

{ "$schema": "...", "resources": [{ "type": "Microsoft.Storage/storageAccounts", "apiVersion": "2021-06-01", "name": "[parameters('storageAccountName')]", "location": "[parameters('location')]", "sku": { "name": "Standard_LRS" }, "kind": "StorageV2", "properties": {} }] }

Bicep (~10 lines)

param storageAccountName string param location string = resourceGroup().location resource storageAccount 'Microsoft.Storage/storageAccounts@2021-06-01' = { name: storageAccountName location: location sku: { name: 'Standard_LRS' } kind: 'StorageV2' }
✅
Bicep is a DSL that transpiles to ARM. When you deploy, Bicep CLI converts it to ARM JSON first. You're still deploying ARM under the hood — Bicep is a better authoring experience, not a different deployment mechanism.
⚠️
Converting ARM → Bicep: The decompiler works but is still maturing. Always review the output before treating it as production-ready.
Bicep File Structure — Building Blocks
Keyword Purpose Example
param Inputs with optional defaults and decorators param location string = resourceGroup().location
var Computed values, calculated once var name = '${prefix}${uniqueString(resourceGroup().id)}'
resource What to actually build resource sa 'Microsoft.Storage/storageAccounts@2021-06-01' = { }
module Reference another .bicep file (relative path) module storage './modules/storage.bicep' = { }
output Return values from deployment output storageId string = storageAccount.id
existing Reference a pre-existing resource not managed in this template resource kv 'Microsoft.KeyVault/vaults@2021-06-01-preview' existing = { name: 'myVault' }
💜
Expert: The existing keyword is hugely useful in real-world work. Reference a Key Vault, subnet, or any other pre-existing resource without managing it in your template.
💜
Expert: Param decorators like @minLength(3) and @maxLength(24) validate at development time — not just at deployment. ARM doesn't have this natively.
Bicep Features — Beyond Basic ARM

🔁 Loops

Create multiple resources from one declaration. Deploy 5 storage accounts with different names using a single resource block.

resource sa 'Microsoft.Storage/storageAccounts@2021-06-01' = [for i in range(0,5): { name: 'storage${i}' }]

❓ Conditionals

Deploy a resource only if a condition is true. Deploy a diagnostic setting only in production environments.

resource diag '...' = if (isProd) { name: 'diagnostics' ... }

📝 String Interpolation

No more verbose concat() functions from ARM. Native string interpolation with ${}.

// ARM: "[concat(parameters('prefix'), '-storage')]" // Bicep: '${prefix}-storage'
Deploying Bicep — GitHub Actions vs Azure Pipelines

GitHub Actions

- name: Deploy Bicep uses: azure/arm-deploy@v1 with: subscriptionId: ${{ secrets.AZURE_SUBSCRIPTION }} resourceGroupName: ${{ secrets.AZURE_RG }} template: ./main.bicep parameters: storageAccountName=mystorageabc

✅ Handles Bicep natively — no pre-compilation needed. azure/arm-deploy@v1 transpiles automatically.

Azure Pipelines

- task: AzureResourceManagerTemplateDeployment@3 inputs: deploymentScope: 'Resource Group' azureResourceManagerConnection: 'MyServiceConnection' resourceGroupName: 'my-rg' location: 'eastus' csmFile: 'main.bicep' deploymentMode: 'Incremental'

⚠️ Note: deploymentMode: 'Incremental' — almost always what you want.

⚠️
Lab Gotcha #1: Module paths must be relative — './modules/storage.bicep' not 'storage.bicep'. Missing the ./ causes file not found errors.
⚠️
Lab Gotcha #2: API version mismatches — if you copy resource declarations from old docs the API version may be deprecated. Read the VS Code linting warnings — they're specific and helpful.

Checkpoints & Parallel Processing

Checkpoints

workflow LongRunningTask { # Step 1 work... Checkpoint-Workflow # ← save state here # Step 2 work... Checkpoint-Workflow # ← save state here # Step 3 fails → resumes from Step 2 # NOT from the beginning }

4-hour runbook fails at hour 3 → without checkpoints: restart from hour 0. With checkpoints: resume from last save. Huge difference at scale.

Parallel Processing

Parallel { 'Restart-VM -Name VM1' ─┐ 'Restart-VM -Name VM2' ─┼─ simultaneous 'Restart-VM -Name VM3' ─┘ } # Then continue sequentially...

Webhooks — External Triggers

External system (GitHub, DevOps, app) | | HTTP POST ▼ https://<server>/token?<token> | ▼ Runbook executes
🔥
Critical: Webhook URL contains an opaque token. Anyone with it can trigger the runbook — treat like a secret. It has an expiry date — when expired, it silently stops working. No error. Set a reminder to rotate it.
⚠️
URL is shown once only at creation. Copy it immediately and store securely. If lost, create a new webhook.

Critical Distinction: ARM/Bicep/CLI create resources — one time. DSC maintains them — continuously. If something drifts, DSC detects and corrects it automatically.
Configuration Drift — The Problem DSC Solves
Day 1: Server deployed correctly via IaC ✓ Day 30: Engineer opens port 8080 "for debugging" "I'll close it after testing" Day 60: Port 8080 still open — nobody noticed Security audit flags it as a finding Day 90: Different engineer installs a package "Just temporary" Server now bears no resemblance to template Result: Security risk, compliance failure, incident waiting to happen

Each individual change seems harmless. The accumulation is the problem. At scale — hundreds of servers — this happens constantly. Manual auditing doesn't scale.

✅
DSC Solution: Detects drift and corrects it automatically. On a schedule. Without human intervention. No ticket raised. Just automatic remediation.
💜
Expert: With Azure Automation + Log Analytics: "Which of my 1,000 servers are non-compliant right now?" — one query, dashboard, real-time. That's what compliance teams need.
DSC Components

📋 Configurations

Idempotent declarative PowerShell scripts. Describe the desired world: "IIS must be installed, service must run, file must exist." Don't describe how — just the end state.

🔧 Resources

PowerShell scripts that compile to MOF files. Know how to check and enforce each thing — WindowsFeature knows how to check if IIS is installed and install it if not.

⚙️ LCM

Local Configuration Manager — client engine on each node. Periodically checks actual state vs desired state. When it finds drift, it remediates automatically.

Structure Breakdown

Configuration block Name of the configuration
Node block Which nodes to configure
Resource blocks The actual desired state
Ensure = 'Present' This thing must exist
Ensure = 'Absent' This thing must NOT exist

LP5 Cheat Sheet — One-Liners

Core Concepts

  • Declarative = describe what, not how → idempotent by design
  • Imperative = step-by-step instructions → may fail on re-run
  • Idempotent = run N times, same result every time
  • Immutable infra = don't patch, discard and redeploy
  • Snowflake server = unique, hand-built, irreproducible
  • Configuration drift = gap between desired and actual state
  • Incremental mode = add/update only, never delete
  • Complete mode = destructive, deletes unlisted resources

Tool Quick Reference

  • ARM = native Azure format, JSON, everything compiles to it
  • Bicep = DSL → transpiles to ARM, less code, same power
  • Azure CLI = imperative, ad-hoc, scripting, queries
  • Azure Automation = scheduled ops, runbooks, not provisioning
  • DSC = continuous state enforcement, detects + fixes drift
  • LCM = client engine on each DSC-managed node
  • MOF file = what DSC config compiles to, applied by LCM
  • nx prefix = Linux DSC resources (nxFile, nxService, nxPackage)
