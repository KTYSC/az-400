DORA Metrics — The 4 Key DevOps Measures

DORA (DevOps Research and Assessment) identified 4 metrics that predict software delivery performance:

📦 Deployment Frequency
How often you deploy to production. Measures throughput.
Elite: Multiple deploys per day
⏱️ Lead Time for Changes
Time from commit to running in production. Measures speed.
Elite: Less than 1 hour
🔧 Mean Time to Restore (MTTR)
Time to recover from a production incident. Measures reliability.
Elite: Less than 1 hour
🐛 Change Failure Rate
% of deployments that cause a production failure. Measures quality.
Elite: 0–15%
🎯
Deployment Frequency + Lead Time = throughput/speed metrics. MTTR + Change Failure Rate = stability/quality metrics. Elite teams excel at BOTH — speed AND stability are not a trade-off with DevOps.
Waterfall vs Agile vs DevOps
ModelDeliveryFeedbackChange
WaterfallOne big release, months/yearsAt the very endExpensive, resisted
AgileIterative sprints, weeksEnd of each sprintEmbraced in backlog
DevOpsContinuous, hours/daysReal-time production dataAutomated, low risk
💡
DevOps doesn't replace Agile — it extends it. Agile improved planning and collaboration. DevOps extends that into deployment and operations with automation and monitoring.
DevOps Key Practices
  • 🔄 Continuous Integration (CI): merge code frequently, automated build + test on every commit
  • 🚀 Continuous Delivery (CD): every build is potentially deployable to production
  • 🏗️ Infrastructure as Code (IaC): manage infra via code, version-controlled
  • 📊 Continuous Monitoring: observability in production, telemetry-driven decisions
  • 🧪 Automated Testing: unit, integration, and e2e tests in the pipeline
  • 🔐 DevSecOps: security integrated throughout, not bolted on
  • 📋 Agile Planning: backlog, sprints, retrospectives

Module 3: Design and Implement Branch Strategies and Workflows
🎯
EXAM FOCUS: Know trunk-based development vs Gitflow vs GitHub Flow trade-offs. Know Azure Repos branch policies — specifically what each policy enforces. Merge strategies: squash vs rebase vs merge commit.
Branching Strategy Comparison
StrategyBranchesRelease CadenceBest ForComplexity
Trunk-Basedmain only (+ short-lived feature branches <1 day)Continuous / multiple/dayHigh-frequency CI/CD, mature teamsLow (with feature flags)
GitHub Flowmain + feature branchesContinuous (on merge)Web apps, SaaS, frequent releasesLow
Gitflowmain, develop, feature, release, hotfixScheduled releasesPackaged software, versioned productsHigh
Release Flowmain + release/ branchesScheduled (branch from main)Microsoft's internal model, large teamsMedium
🎯
Trunk-based development is the approach used by elite DevOps performers. Short-lived branches (< 1 day) merged to main. Requires strong CI, automated testing, and feature flags. Gitflow is for scheduled releases, not continuous delivery.
Gitflow — Branch Types
  • main: production-ready code only. Tags mark releases.
  • develop: integration branch for features. CI runs here.
  • feature/: branch off develop, merge back to develop when done.
  • release/: branch off develop when ready to release. Only bug fixes.
  • hotfix/: branch off main for urgent production fixes. Merge to both main AND develop.
⚠️
Gitflow gets complex fast. Long-lived branches = merge conflicts = pain. If you're doing CI/CD, Gitflow works against you. Use it only for products with versioned, scheduled releases.
Merge Strategies
StrategyHistoryUse When
Merge CommitPreserves all commits + adds merge commitFull history needed, traceability important
Squash MergeAll feature commits → 1 clean commit on mainClean main history, messy feature branch WIP commits
RebaseReplays commits on top of main, linear historyClean linear history, no merge commits
🎯
Squash merge = one commit on main per feature. Clean history. Rebase = replays commits, linear history, rewrites SHA hashes — dangerous on shared branches. Never rebase public/shared branches.
Azure Repos Branch Policies

👥 Minimum Reviewers

Require N approvals before a PR can merge. Prevents self-merging. Can require fresh approval when new commits are pushed.

✅ Build Validation

Trigger a pipeline run on the PR. PR cannot merge if build fails. Enforces CI on every change.

🔗 Work Item Linking

Require at least one linked work item on the PR. Ensures every change traces back to a planned work item.

📝 Comment Resolution

All PR comments must be resolved before merge. Prevents merging with unaddressed review feedback.

🔒 Merge Strategy

Enforce a specific merge strategy (squash, rebase, merge commit) — or allow the author to choose.

🔄 Up-to-date Branch

Require PR branch to be up to date with main before merging. Prevents merge-order issues.

🎯
Branch policies are enforced even on repo admins (unless bypassed explicitly). Build validation is the policy that runs your CI pipeline on every PR — the most important policy for quality gates.

CODEOWNERS

CODEOWNERS file automatically assigns reviewers based on which files are changed in the PR.

# GitHub: .github/CODEOWNERS # Format: path @owner(s) *.js @frontend-team /src/api/ @backend-team /docs/ @tech-writers *.bicep @infra-team /security/ @security-team @lead-dev
🎯
CODEOWNERS is a GitHub feature. In Azure DevOps, required reviewers per path are configured as branch policies. Both achieve the same goal — right person reviews right code.
💡
Real world: CODEOWNERS prevents the "anyone can approve anything" anti-pattern. The security team is auto-added to any PR touching /security/ — no one can forget to include them.
Code Review Best Practices
Review for logic, security, and maintainability — not just style. Use linters/formatters to automate style so reviews focus on what matters.
Request changes for purely personal preference ("I would have done it differently"). Be specific — explain WHY something should change.
Keep PRs small — <400 lines changed. Small PRs get better reviews, merge faster, and are easier to roll back.
Open a "monster PR" with 3,000 lines changed. Reviewers rubber-stamp large PRs — they can't meaningfully review them. Bugs hide in big PRs.
💡
Research shows that review effectiveness drops sharply after 400 lines. Above 400 lines, defect detection rate falls from ~65% to <20%. Small PRs are a quality strategy, not just a team preference.

Module 5: Explore Git Hooks
🎯
EXAM FOCUS: Know client-side vs server-side hooks and their trigger points. Know that client-side hooks can be bypassed (--no-verify). Server-side hooks cannot be bypassed by the developer.
Client-Side Hooks

Run on the developer's machine. Stored in .git/hooks/ — NOT committed to the repo by default.

HookTriggersCommon Use
pre-commitBefore commit is createdRun linter, formatter, unit tests
commit-msgAfter commit message enteredValidate message format (e.g., Conventional Commits)
pre-pushBefore pushing to remoteRun full test suite, security scan
post-commitAfter commit createdNotifications, local logging
prepare-commit-msgBefore commit message editor opensAuto-populate message template
⚠️
Client-side hooks can be bypassed with git commit --no-verify. Don't rely on them as security controls — use server-side hooks or pipeline checks for enforcement.
Server-Side Hooks

Run on the Git server (Azure Repos, GitHub). Cannot be bypassed by the developer.

HookTriggersCommon Use
pre-receiveBefore accepting any pushReject commits with large files, enforce policies
updateOnce per branch being updatedBranch-specific policies
post-receiveAfter push fully processedNotifications, CI trigger, deploy hooks
🎯
In Azure DevOps, server-side enforcement is done through branch policies and service hooks — not raw Git hooks. Azure DevOps Service Hooks trigger on events (push, PR created, build completed) and call external services.
💡
Husky = popular npm package to manage and share client-side hooks in a project. Hooks stored in .husky/ and checked into the repo — solves the "hooks aren't shared" problem for JS/Node teams.

Module 6: Plan Foster Inner Source
What is Inner Source?

Inner Source = applying open-source development practices within a private organization.

  • Code is visible across all internal teams (not siloed per squad)
  • Anyone can submit PRs to improve shared components
  • Reduces duplicate work across teams
  • Spreads expertise and best practices organically
  • Creates a culture of collaboration, not ownership silos
🎯
Inner source is the "S" in CALMS (Sharing). It requires: discoverable repos, good documentation, welcoming contribution guides, and trusted committers — senior devs who own the quality of the shared component.
Inner Source Requirements
ElementPurpose
README.mdWhat the repo does, how to build, how to use it
CONTRIBUTING.mdHow to submit issues and PRs, coding standards
CODEOWNERSWho reviews PRs — the trusted committers
CODE_OF_CONDUCT.mdCommunity behavior standards
Issue templatesStructured bug reports and feature requests
💡
Trusted Committer = the gatekeeper of a shared repo. Reviews external PRs, maintains quality, mentors contributors. Critical role for inner source success — without it, shared repos become graveyards of unreviewed PRs.

Git LFS — Large File Storage

Git LFS stores large binary files outside the Git object store, replacing them with small pointer files in the repo.

  • Use for: images, videos, audio, 3D models, ML model files, compiled binaries
  • Pointer file stays in repo (~134 bytes). Actual file in LFS store.
  • Git history stays fast — large files don't bloat clone/fetch
  • Requires: Git LFS client installed, LFS-enabled server (GitHub, Azure Repos both support it)
# Track all PNG files with LFS git lfs track "*.png" git add .gitattributes git add image.png git commit -m "Add image via LFS"
🎯
"Team is experiencing slow clone times due to large binary files" → answer = Git LFS. It solves the repo bloat problem caused by checking in large binaries.

Module 8: Identify Technical Debt
🎯
EXAM FOCUS: SonarQube / SonarCloud as the primary code quality tool. Know what each rating (reliability, security, maintainability) measures. Know code coverage and its limits as a metric.
What is Technical Debt?

Technical debt = the implied cost of rework caused by choosing an easy solution now instead of a better approach that would take longer.

Debt TypeExamples
Code DebtDuplicated code, poor naming, long methods, hardcoded values
Test DebtLow code coverage, missing integration tests, flaky tests
Architecture DebtMonolith that should be services, wrong DB choice, tight coupling
Documentation DebtOutdated docs, no API docs, missing runbooks
Infra DebtOld OS versions, unpatched dependencies, manual deployments
💡
Some debt is deliberate (time-to-market trade-off) and acceptable — IF you document it and pay it back. Unmanaged, accidental debt compounds and eventually paralyzes a team. Track it in your backlog like any other work.
SonarQube / SonarCloud

SonarQube (self-hosted) / SonarCloud (cloud) = the industry-standard code quality analysis platform.

RatingWhat It MeasuresScale
ReliabilityBugs — code that will likely behave incorrectlyA–E (A = no bugs)
SecurityVulnerabilities — exploitable code weaknessesA–E (A = no vulns)
MaintainabilityCode smells — debt ratio. Time to fix / dev timeA–E based on debt ratio
🎯
Quality Gate = SonarQube policy that blocks the build/PR if metrics fall below thresholds. e.g., "Fail if new code coverage < 80% or Reliability Rating < A". Quality Gates enforce code quality automatically in CI.
💡
Cognitive Complexity = SonarQube metric for how hard code is to understand (vs cyclomatic complexity which just counts branches). Lower = easier to read and maintain.
Code Coverage — Metrics and Limits

Code coverage = percentage of production code lines/branches executed by tests.

Coverage TypeMeasures
Line CoverageWhich lines were executed
Branch CoverageWhich code branches (if/else) were taken
Function CoverageWhich functions were called
⚠️
Coverage ≠ quality. 100% line coverage can still have zero assertions. A test that calls every function but never checks the output gives 100% coverage and zero bug detection. Coverage is a floor, not a ceiling.
🎯
Industry target: 80% code coverage is a common minimum. But 80% meaningful coverage > 100% coverage with no assertions. The exam tests the concept, not a specific number.
