Module 1: Introduction to Deployment Patterns
🎯
EXAM WEIGHT: LP4 patterns appear in pipeline scenario questions. Know the difference between deploy and release, understand each pattern's risk profile, and know which Azure services enable each pattern.
Deploy vs Release — Critical Distinction
ConceptDefinitionExample
DeployPut code into an environment (technical act)Push v2 to the staging slot
ReleaseMake that code available to users (business act)Swap staging → production
🎯
Feature flags decouple deploy from release. You can deploy code at 2am (dark, off) and release it at 9am (toggle on) — no new deployment needed. This is the superpower of progressive delivery.

Traditional approach: deploy = release (risky). Modern approach: deploy early, release when ready.

Why Deployment Patterns Exist

Every deployment pattern solves the same core problem: how do we get new code to users while minimizing risk?

  • Big bang deployments = maximum risk, maximum blast radius
  • Progressive patterns = reduce blast radius, enable fast rollback
  • Each pattern trades off complexity vs risk reduction
  • Right pattern depends on: team size, traffic volume, tolerance for complexity
💡
Netflix, Google, and Amazon deploy hundreds of times per day using these patterns. Zero-downtime is not a luxury — it's an engineering discipline.

Deployment Rings

Rings = concentric audience circles. Ship to inner rings first, monitor, expand outward:

0Dev / EngineeringInternal only
1Early AdoptersBeta users / Insiders
2Limited Users~10% of traffic
3All UsersGeneral availability
🎯
Microsoft's Windows and Edge use deployment rings internally. Ring 0 = Microsoft employees. Ring 1 = Windows Insiders. Ring 3 = general public. The principle scales to any app.
Rollback vs Roll Forward
StrategyApproachWhen to Use
RollbackRevert to previous known-good versionCritical bug, data corruption risk, unclear root cause
Roll ForwardFast-path a hotfix deployment forwardQuick fix available, database migrations already ran, rollback too complex
⚠️
Database schema migrations make rollback dangerous — if new code added a column and old code doesn't know about it, rolling back can cause data errors. Plan for backward-compatible migrations to keep rollback viable.

Azure App Service Deployment Slots

Deployment Slots are the Azure-native blue-green mechanism for App Service.

  • Each slot = a separate instance of your app with its own URL
  • Standard tier and above support multiple slots
  • Deploy to staging slot, test, then swap to production
  • Swap = warm-up routing. Azure sends requests to new slot until it responds, then flips traffic
🎯
Slot settings (sticky settings) — app settings and connection strings marked as "slot setting" stay with the slot and DON'T swap. Perfect for environment-specific settings (staging DB vs prod DB). Settings NOT marked as sticky WILL swap with the code.
Azure CLI
az webapp deployment slot swap --name myapp --resource-group myRG --slot staging --target-slot production
✓ Swap completed. Production now running v2.
Feature Toggles — 4 Types
🎯
The exam tests the 4 feature toggle types. Know what each one is used for — not just that feature flags exist.

🚀 Release Toggle

Deploy incomplete features hidden behind a flag. Enable when ready — no new deployment. Most common type. Short-lived by design.

Short-lived

⚙️ Ops Toggle

Kill switch for features in production. If a feature degrades performance, ops can disable it instantly without a deployment. Circuit breaker pattern.

Long-lived

🧪 Experiment Toggle

A/B testing toggle. Route cohorts of users to different code paths, collect telemetry, decide winner. Removed after experiment concludes.

Short-lived

🔒 Permission Toggle

Expose features to specific user segments — premium users, beta testers, specific regions. Long-lived; tied to user tier/subscription/role.

Long-lived

⚠️
Toggle debt = stale flags that never get cleaned up. Each active flag = a code branch that must be tested. 50 old flags = exponential test complexity. Set expiry dates and remove flags after rollout.
💡
Azure App Configuration = Azure-native feature flag service. Also: LaunchDarkly (industry standard), Unleash (open source), or simple config values.

Dark Launching

Dark launching = run the new code path silently alongside the old one — real production traffic, but results are hidden from users.

  • Both old and new code process the same request
  • New code's response is logged/compared but not shown
  • Validate correctness, performance, and stability under real load
  • Zero user impact — any bugs are silent
💡
Real example: GitHub used dark launching to test their new Git backend — ran it on every real request for months comparing outputs before switching. No users ever saw the new system until it was proven.

🎯
The exam trap: canary = users experience the new version (visible). Dark launch = new version runs in the background silently. If a question says "no users are affected by bugs" → dark launch.

Module 4: Implement A/B Testing and Progressive Exposure Deployment
A/B Testing

A/B Testing is a controlled experiment — two variants shown to different user groups, measured against a business metric.

  • Variant A: control (current experience)
  • Variant B: treatment (new experience)
  • Randomly assign users to variants — randomization is critical
  • Measure: conversion rate, click-through, revenue, engagement
  • Run until statistical significance achieved (p < 0.05)
  • Declare winner, remove losing variant, clean up toggle
⚠️
Never end an A/B test early because one variant is leading. Statistical significance requires enough samples — early stopping introduces bias. A test "looks like it's winning" at day 2 and loses by day 14 all the time.

A/B = business experiment (which button color converts better?). Canary = engineering risk management (is this new version stable?). Different goals, different metrics, different timescales.

Azure DevOps Service Connections

Service Connections = authenticated connections from Azure DevOps to external services, stored securely at project or org level.

☁️ Azure Resource Manager

Auth to Azure subscriptions. Supports Workload Identity Federation (recommended), Service Principal + secret, or Managed Identity.

🐙 GitHub

Auth to GitHub repos for pipeline triggers, checkout, or package publishing. Uses PAT or OAuth App.

🐳 Docker Registry

Push/pull container images. Supports Docker Hub, Azure Container Registry (ACR), and any generic v2 registry.

🎯
Service connections can be scoped to specific resource groups or subscriptions — principle of least privilege. Never use a subscription-owner service connection when only one RG needs access.

Module 6: Manage Application Configuration Data
🎯
EXAM FOCUS: Azure App Configuration vs Azure Key Vault — know what each stores and when to use which. Know Key Vault soft-delete + purge protection. Know App Configuration labels, snapshots, and Key Vault references.
Azure App Configuration

Azure App Configuration = centralized, managed service for app settings and feature flags.

  • Key-value pairs with optional labels (dev, staging, prod, region)
  • Feature flags — built-in management with targeting, rollout %
  • Key Vault references — store a reference to a KV secret (not the secret itself)
  • Snapshots — point-in-time copy of configuration for rollbacks
  • Change notifications — push config changes to apps without restart
  • Hierarchical keys: App:Database:ConnectionString
🎯
Labels = variant selector. Same key, different labels = different values per environment. ConnectionString with label dev → dev DB. Same key with label prod → prod DB. One config store, multiple environments.
Azure Key Vault

Azure Key Vault = secure storage for secrets, keys, and certificates. Not a config store — a secrets store.

  • Secrets: connection strings, passwords, API keys
  • Keys: cryptographic keys for encryption operations (HSM-backed optional)
  • Certificates: SSL/TLS certificate lifecycle management
🎯
Soft Delete = deleted vault/secret moved to "deleted" state for retention period (7-90 days). Can be recovered. Purge Protection = even after soft-delete, cannot be permanently deleted until retention expires — even by admins. Enable both for production Key Vaults.
🚨
Purge protection + soft delete = your last line of defense against accidental or malicious Key Vault deletion. Without purge protection, anyone with delete permissions can permanently destroy a vault.
App Configuration vs Key Vault — When to Use Which
StoreWhat to Put HereWhat NOT to Put Here
App ConfigurationFeature flags, app settings, environment labels, non-sensitive config values, Key Vault referencesPasswords, connection strings, API keys, certificates
Key VaultSecrets (passwords, connection strings, API keys), encryption keys, TLS certificatesNon-sensitive app settings, feature flags, UI config
💡
Best practice pattern: Store all config in App Configuration. For sensitive values, store a Key Vault reference in App Configuration — it just holds a pointer (URI) to the actual secret in Key Vault. App fetches secret at runtime via the reference. One place to look, secure storage for secrets.
Key Vault Access — Policies vs RBAC
ModelHow It WorksRecommendation
Access PoliciesVault-level permission model. Grant identities get/set/list/delete on secrets/keys/certs separately.Legacy. Fine-grained but harder to manage at scale.
Azure RBACStandard Azure role assignments. Built-in roles: Key Vault Reader, Secrets User, Secrets Officer, Administrator.✅ Recommended. Consistent with rest of Azure RBAC.
🎯
Key Vault RBAC model = Key Vault Secrets User (read secrets only) vs Key Vault Secrets Officer (create/update/delete). Pipelines should get Secrets User — read only, no write.
Variable Groups in Azure Pipelines

Variable Groups = named sets of variables reusable across pipelines.

  • Stored in Azure DevOps Library
  • Link a variable group to a Key Vault — sync secrets automatically at pipeline run time
  • Mark variables as secret — masked in logs, not passed between stages by default
  • Reference in YAML: group: MyVariableGroup
variables: - group: Production-Secrets # links to Key Vault steps: - script: echo "Connecting to DB..." env: DB_CONN: $(SqlConnectionString) # fetched from KV
🎯
Variable groups linked to Key Vault require a service connection with access to the vault. The pipeline identity needs at least Key Vault Secrets User role on the vault.
