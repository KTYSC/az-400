What is DevSecOps?

DevSecOps = DevOps + Security baked in from day one — not bolted on at the end.

  • Security is a shared responsibility across Dev, Ops, and Security teams
  • Shift-left = find vulnerabilities earlier in the SDLC when they're cheaper to fix
  • Automate security checks in CI/CD pipelines
  • Continuous compliance monitoring, not periodic audits
🎯
The cost-of-defect model: fixing a bug costs $80 in design, $240 in coding, $960 in testing, $7,600 in production. This is the business case for shift-left.

🔍 SAST

Static Application Security Testing — analyzes source code without running it. CodeQL, SonarQube. Catches logic flaws, injection risks.

🌐 DAST

Dynamic Application Security Testing — tests a running app. OWASP ZAP. Finds runtime vulnerabilities like XSS, SQL injection.

📦 SCA

Software Composition Analysis — scans third-party dependencies for known CVEs. Dependabot, WhiteSource, Mend.

Azure Key Vault in DevOps

✅

  • Key Vault task: Azure Pipelines native integration
  • Variable Groups: link to Key Vault, sync secrets automatically
  • Managed Identity: pipeline authenticates without storing credentials
🎯

OSS Risk Management

Why enterprises care about OSS licenses:

  • Using GPL library in a commercial product → must open-source your product
  • License violations = legal liability, IP exposure
  • Abandoned packages = unpatched CVEs forever

CodeQL — Code as a Database

CodeQL treats your source code as a queryable database. Write queries to find vulnerability patterns across the entire codebase.

  • Supports: C/C++, C#, Java, JavaScript/TypeScript, Python, Go, Ruby
  • Finds: SQL injection, XSS, path traversal, deserialization flaws
  • Integrated into GitHub Advanced Security as code scanning
  • Runs on every pull request — blocks merge if critical issues found
🎯
The exam phrase: "CodeQL treats source code as a queryable database" — this exact framing shows up in questions. CodeQL ≠ runtime scanner. It's static analysis.
name: CodeQL Analysis on: [push, pull_request] jobs: analyze: runs-on: ubuntu-latest steps: - uses: github/codeql-action/init@v3 with: languages: javascript - uses: github/codeql-action/analyze@v3

CVSS — Vulnerability Scoring

CVSS (Common Vulnerability Scoring System) — industry standard 0-10 scale:

ScoreSeverityAction
9.0 – 10.0CriticalPatch immediately / emergency change
7.0 – 8.9HighPatch within 7 days
4.0 – 6.9MediumPatch within 30 days
0.1 – 3.9LowPatch in next release cycle
⚠️
CVSS base score doesn't account for your environment. A Critical CVE in a library you don't use is effectively zero risk. Use exploitability + asset criticality to prioritize.
Security Bug Bar

A security bug bar defines the minimum severity threshold that blocks a release:

  • E.g.: "No Critical or High CVEs may ship to production"
  • Enforced via pipeline gates — fail the build if thresholds exceeded
  • Used with CVSS scores from SCA and SAST tools
  • Different bars per environment (dev/staging/prod)
🎯
The security bug bar is how organizations operationalize "zero Critical vulns in prod" — it's a policy enforced in the pipeline, not a manual review process.
💡
Real world: Teams create exceptions/waivers for accepted risk with business owner sign-off. The bug bar is the default; exceptions prove the rule.

Azure Policy

Azure Policy enforces organizational standards and assesses compliance at scale.

  • A policy rule = condition (if) + effect (then)
  • Assigned to management groups, subscriptions, or resource groups
  • Evaluated continuously — not just at creation time
EffectWhat Happens
DenyBlock the resource from being created/modified
AuditAllow but log non-compliant resources
DeployIfNotExistsAuto-deploy a related resource if missing
AppendAdd additional fields to the resource request
ModifyAdd/replace/remove properties during create/update
🎯
Initiative = a collection of policies grouped together for a compliance goal (e.g., "PCI-DSS" initiative bundles 50 individual policies). Assign one initiative instead of 50 policies.
Resource Locks

Resource Locks prevent accidental deletion or modification — even by admins with full RBAC permissions.

Lock TypeEffect
CanNotDeleteRead and modify allowed. Delete blocked.
ReadOnlyOnly reads allowed. Modify AND delete blocked.
🚨
Locks override RBAC. Even an Owner-role user cannot delete a locked resource without first removing the lock. Locks live at resource, resource group, or subscription scope and inherit downward.
💡
Real world: Apply CanNotDelete to production Key Vaults and storage accounts. Prevents "oops I deleted prod" incidents that have ended careers.

Unified security management across GitHub and Azure DevOps pipelines.

  • Connects to GitHub repos and Azure DevOps projects via connector

  • Provides DevOps security posture dashboard
🎯
Defender for Identity — 3 components: Sensor (installed on DCs), Cloud Service (Azure-hosted), Management Console (portal UI). Detects lateral movement, pass-the-hash, golden ticket attacks.
GitHub Advanced Security in Azure DevOps

Microsoft Security DevOps (MSDO) extension — runs in Azure Pipelines:

  • Integrates: Bandit, BinSkim, ESlint, TemplateAnalyzer, Terrascan, Trivy

Module 1: Explore Package Dependencies
🎯
EXAM WEIGHT: LP7 topics appear in pipeline and build questions. Know Azure Artifacts feeds deeply — feeds, upstream sources, views, and package types are all exam targets.
What Are Package Dependencies?

Modern apps don't build from scratch — they assemble from packages. Understanding the dependency graph is critical for security, compliance, and build stability.

  • Direct dependency: explicitly declared in your manifest (package.json, .csproj, pom.xml)
  • Transitive dependency: a dependency of your dependency — not declared, but pulled in
  • Dependency hell: conflicting version requirements across the tree
⚠️
Transitive vulnerabilities are the hardest to catch. A critical CVE in a deep transitive dep won't show up in your manifest — that's why SCA tools parse the full dependency tree.
Package Manifest Files
EcosystemManifest FileLock File
Node.js / npmpackage.jsonpackage-lock.json
.NET / NuGet.csproj / packages.configpackages.lock.json
Java / Mavenpom.xml—
Python / piprequirements.txtPipfile.lock
Java / Gradlebuild.gradlegradle.lockfile
💡
Lock files pin exact transitive versions for reproducible builds. Always commit lock files to source control for application repos. Libraries should NOT commit lock files — let consumers resolve.
Dependency Management Best Practices
Use version ranges (e.g., ^1.2.0) in library manifests — let consumers pin their own versions.
Use wildcards (*) in production app manifests — unpredictable builds, security risk from auto-updating to malicious versions.
Use upstream sources in Azure Artifacts to cache public packages — ensures availability even if npm registry goes down.
Pull directly from public registries in CI/CD — if the registry is unavailable or a package is yanked, your pipeline breaks.

Azure Artifacts — Core Concepts

Azure Artifacts = a universal artifact repository hosted in Azure DevOps. One place for all package types.

  • Feed: a container that holds packages. You configure permissions and upstream sources per feed.
  • Org-scoped feed: visible to all projects in the org
  • Project-scoped feed: visible only within the project
  • Upstream source: a feed can proxy packages from public registries (npmjs.com, nuget.org, PyPI)
🎯
When a feed has an upstream source configured, requesting a package that isn't in the feed will pull it from upstream and cache a copy. Future requests hit the cache — no internet dependency.

Feed Views — Promotion Pipeline

Views are a way to promote packages through quality gates without copying them:

Module 3: Migrate, Consolidate and Secure Artifacts
Why Consolidate Artifact Stores?

Many orgs have packages scattered across: NuGet feeds, local file shares, Artifactory instances, Jenkins build outputs. Consolidating to Azure Artifacts gives:

  • Single source of truth for all packages
  • Uniform access control (Azure AD / RBAC)
  • Audit trail for every package download/publish
  • Retention policies to clean up old versions
  • Upstream sources to replace fragmented external feeds
Securing Artifacts

Package security layers in Azure Artifacts:

  • Feed permissions: Owner, Contributor, Collaborator, Reader roles
  • Package immutability: once published, a version cannot be overwritten — prevents tampering
  • Delete vs Unlist: delete removes from feed entirely; unlist hides from search but still installable with exact version
  • Upstream blocking: disable upstream sources to prevent packages from public registries reaching dev environments
🎯
Delete vs Unlist: Unlist = hides from browse/search but version is still installable if you specify it exactly. Delete = permanently gone. Use Unlist to deprecate; Delete only when appropriate.
Artifact Retention Policies

Retention policies automatically clean up old pipeline artifacts and package versions:

  • Set max versions to retain per package
  • Configure days to retain pipeline runs
  • Protects storage costs — old builds accumulate fast
  • Minimum retention = 1 version (can never auto-delete all)
⚠️
Retention policy trap: If a build artifact is associated with a release that hasn't been deleted, the retention policy will NOT delete that artifact — releases protect their artifacts. You must delete the release first.

Module 4: Implement a Versioning Strategy

Semantic Versioning (SemVer)

Format: MAJOR.MINOR.PATCH

ComponentWhen to IncrementExample
MAJORBreaking change — consumers must update code2.0.0
MINORNew backward-compatible feature added1.3.0
PATCHBackward-compatible bug fix only1.2.7

Pre-release identifiers: 1.0.0-alpha → 1.0.0-beta → 1.0.0-rc.1 → 1.0.0

🎯
Version 0.x.x = initial development, anything may change. Version 1.0.0 = public API declared stable. Starting at 0 is intentional — no stability guarantees.
Versioning Strategies for Pipelines
  • Date-based: 2024.04.03.1 — easy to understand, no semantic meaning
  • GitVersion: auto-calculates SemVer from git history and branch names
  • Build counter: append pipeline run number — $(Build.BuildNumber)
  • Commit SHA: short SHA for traceability (not SemVer)
💡
In real projects: use GitVersion for library packages (proper SemVer), use date + build counter for deployment artifacts. Libraries need semantic meaning; deployments need traceability.
# GitVersion task in Azure Pipelines - task: GitVersion@5 inputs: versionSpec: '5.x' # Version available as $(GitVersion.SemVer)

GitHub Packages = package registry integrated directly with GitHub repositories.

  • Supports: npm, NuGet, Maven, RubyGems, Docker/OCI containers

SLI / SLO / SLA — The Reliability Framework

📏 SLI — Service Level Indicator

A specific metric measuring system behavior. Example: "request latency at p99", "availability percentage", "error rate". Raw measurement.

🎯 SLO — Service Level Objective

Target value for an SLI. Internal commitment. Example: "p99 latency < 200ms", "availability > 99.9%". If SLO is breached → burn error budget.

📄 SLA — Service Level Agreement

External, contractual commitment to customers. Usually includes financial penalties. Always less strict than internal SLO — SLO is your buffer.

🎯
Error budget = 1 - SLO. If SLO = 99.9% availability → error budget = 0.1% downtime = 8.7 hours/year. Spend error budget on risky deployments; when depleted → freeze releases.

Module 3: Share Knowledge Within Teams
Azure DevOps Wikis

Two types of wikis in Azure DevOps:

  • Project Wiki: Created by Azure DevOps, stored as a Git repo in the project. Markdown-based. Version-controlled. Default choice.
  • Published Code Wiki: Publishes a folder in an existing repo as a wiki. Docs live alongside code — great for API documentation.
🎯
Project Wiki = standalone wiki repo. Published Code Wiki = points to a folder in a code repo. If you want docs next to code → Published Code Wiki. If you want a shared team knowledge base → Project Wiki.
GitHub Discussions & Team Collaboration

Beyond code — team knowledge sharing tools:

  • GitHub Discussions: Forum-style conversations within a repo. Q&A, announcements, ideas. Separate from Issues (bug/feature tracking).
  • GitHub Pages: Host static docs sites from a repo branch. Commonly used for project documentation.
  • ADO Team Dashboards: Visible KPIs, sprint progress, and alerts for the whole team — shared visibility.
💡
Discussions vs Issues: Issues = actionable work items with close/reopen lifecycle. Discussions = ongoing conversation with no required resolution. Don't use Issues for "open-ended questions."
Inner Source and Knowledge Culture

Inner Source = applying open-source collaboration practices within a company:

  • Code is visible across teams (not just your squad)
  • Anyone can submit PRs to improve shared components
  • Reduces silos, avoids reinventing wheels
  • Requires discoverable repos + welcoming contribution guides
💡
Teams with strong inner source culture have 30-50% less duplicated code across the org. The key enabler: searchable, well-documented repos + a culture that rewards contributions outside your team.

Continuous Monitoring in DevOps

Continuous monitoring integrates observability into every stage of the DevOps loop — not just production:

  • Pre-production: Load tests, synthetic tests, performance benchmarks in CI
  • Deployment gates: Auto-query App Insights after deployment — fail release if error rate spikes
  • Post-deployment: Canary analysis, progressive rollout with telemetry validation
  • Production: Real user monitoring, synthetic availability tests
🎯
Release gates in Azure Pipelines can query Application Insights / Azure Monitor automatically. If error rate exceeds threshold post-deploy → gate blocks progression or triggers rollback.
Feature Flags + Analytics

Feature flags enable controlled rollouts with telemetry-driven decisions:

  • Enable feature for 5% of users → measure conversion, error rate, performance
  • If metrics OK → expand to 25% → 100%
  • If metrics bad → toggle off instantly — no deployment needed
  • A/B test variants with statistical significance
💡
Tools: Azure App Configuration (feature flags + config management), LaunchDarkly (advanced targeting + analytics), Unleash (open source). App Configuration integrates natively with Azure DevOps pipelines.
🎯
Azure App Configuration = the Azure-native feature flag and centralized configuration service. Integrates with Key Vault for secrets. Exam tests: what service manages feature flags in Azure → App Configuration.
Feedback Integration with Work Items

Closing the feedback loop — from monitoring to actionable work items:

  • App Insights can auto-create ADO work items from exceptions
  • Smart Detection creates work items for anomaly clusters
  • Azure Monitor alerts can trigger Logic Apps → create GitHub Issues / ADO Bugs
  • ITSM integration: ServiceNow, Jira, PagerDuty via action groups
💡
Best practice: configure exception grouping in App Insights so 1000 instances of the same error = 1 work item, not 1000. Without grouping, your backlog drowns in noise.

🎯

Azure Monitor Alert Rules

Alert rule = Signal + Condition + Action Group

Alert TypeSignal SourceUse Case
Metric AlertAzure Monitor MetricsCPU > 80%, response time > 500ms
Log AlertLog Analytics / KQL queryError count > 50 in 5 min
Activity Log AlertAzure Activity LogVM deleted, policy changed, role assigned
Smart DetectionApplication InsightsAnomaly in failure rate or performance
🎯
Activity Log Alerts fire on management plane operations (someone deleting or modifying resources). Metric/Log Alerts fire on data plane telemetry (app behavior). Know which to use for "alert when a VM is deleted."
Action Groups

Action Group = a collection of notification and action channels triggered by an alert.

Notification types:

  • 📧 Email / SMS / Push / Voice
  • 🔗 Webhook / Secure Webhook
  • ⚡ Azure Function
  • 🔄 Logic App
  • 📋 ITSM connector (ServiceNow, Jira)
  • 🔧 Automation Runbook
  • 🚨 PagerDuty / OpsGenie
💡
Action groups are reusable — attach the same "PagerDuty + Slack + ServiceNow" action group to 20 different alert rules. Change the action group once to update all alerts.
🎯
The exam tests: "Which component of an alert defines what happens when it fires?" → Action Group. The alert rule defines WHEN; the action group defines WHAT.
Blameless Retrospectives

After an incident, a blameless retrospective (post-mortem) focuses on what happened, not who did it:

  • Assume everyone acted with good intentions and best available information
  • Focus on system failures, not individual failures
  • Ask "how could the system have prevented this?" not "who made the mistake?"
  • Action items are systemic improvements: better alerts, runbooks, automation
  • Written post-mortem shared publicly across engineering
💡
Google's SRE book popularized blameless post-mortems. Core insight: if people fear blame, they hide incidents and problems fester. Psychological safety → more incidents reported → faster system improvement.
Just Culture

Just culture is a framework distinguishing between types of errors:

Error TypeResponse
Human Error (honest mistake)Console and support — system redesign to prevent recurrence
At-Risk Behavior (shortcut)Coach and guide — remove incentives for the shortcut
Reckless Behavior (knowing violation)Punitive action — this is the rare exception
🎯
Just Culture ≠ no accountability. Most errors are human/at-risk and need coaching + system fixes, NOT punishment. Only reckless/deliberate violations warrant punitive action. The exam tests this distinction.
